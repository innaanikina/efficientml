### Результаты экспериментов

Все эксперименты прогнаны на NVIDIA A100 80GB PCIe (CUDA 12.3) на модели `unsloth/Llama-3.2-1B-Instruct`. Бенчмарки сравнивают три варианта квантизации:
- **rowwise_int4** - симметричная квантизация по строкам, один scale на строку весов
- **rowwise_int4_asym** - асимметричная (signed) квантизация по строкам со scale + zero-point
- **rowwise_int4_gptq** - GPTQ с group_size=128 и act_order

Все варианты квантизации выполняются для всех 113 `nn.Linear` слоёв модели. Llama-3.2-1B имеет `tie_word_embeddings=True`, то есть `lm_head.weight` и `embed_tokens.weight` указывают на один и тот же тензор. Когда «эмбеддинг квантован», `embed_tokens` подменяется на `TiedQuantizedEmbedding` и читает строки из того же int4-буфера, что и `lm_head`.

#### 1. Сжатие весов

| Кернел | Эмбеддинг (= lm_head) квантован | Linear weights, MiB | Сжатие | Linear-слоёв квантовано |
|---|---|---|---|---|
| baseline (fp16) | - | 2357.00 | 1.00x | - |
| rowwise_int4 | да | 590.21 | **3.99x** | 113 / 113 |
| rowwise_int4 | нет | 965.72 | 2.44x | 112 / 113 |
| rowwise_int4_asym | да | 592.14 | 3.98x | 113 / 113 |
| rowwise_int4_asym | нет | 967.16 | 2.44x | 112 / 113 |
| rowwise_int4_gptq | да | 610.16 | 3.86x | 113 / 113 |

Замечание: разница между «да» и «нет» здесь - это именно 501 MiB общего тензора `lm_head` ≡ `embed_tokens` (`128256 x 2048` в fp16). Когда мы оставляем его в fp16, мы оставляем и эмбеддинг входа, и матрицу выхода - поэтому экономия пропадает не только за счёт «выходной проекции», а в первую очередь за счёт того, что эмбеддинг словаря в 128256 строк остаётся непережатым. Без квантизации этого тензора 4x сжатие в принципе недостижимо, так как он один - 21% всех Linear-весов.

#### 2. Качество (perplexity на WikiText-2, 8192 токенов)

| Кернел | Эмбеддинг квантован | Perplexity | Δ к baseline | Ratio к baseline |
|---|---|---|---|---|
| baseline fp16 | - | **20.50** | - | 1.00x |
| rowwise_int4 | да | 55.08 | +34.58 | 2.69x |
| rowwise_int4 | нет | 52.57 | +32.07 | 2.56x |
| rowwise_int4_asym | да | 36.41 | +15.91 | 1.78x |
| rowwise_int4_asym | нет | 34.39 | +13.89 | 1.68x |
| rowwise_int4_gptq | да | **22.91** | +2.41 | 1.12x |

Выводы по качеству:
- Симметричная rowwise-квантизация даёт серьёзную деградацию (PPL x2.7) - распределение весов не нулецентричное, и потеря бина под знак критична.
- Асимметричная (со scale + zero-point) уже значительно лучше, но всё равно теряет ~75% по PPL.
- GPTQ практически восстанавливает качество fp16 (ΔPPL ≈ 2.4) - за счёт калибровочного решения задачи минимизации ошибки + group_size=128 + act_order.
- Сохранение эмбеддинга в fp16 улучшает PPL лишь на ~2 пункта (для sym/asym), что значительно меньше эффекта от sym → asym → GPTQ.

#### 3. Скорость инференса

| Кернел | Эмбеддинг квантован | Tokens/s | vs baseline |
|---|---|---|---|
| baseline fp16 | - | **73.0** | 1.00x |
| rowwise_int4 | да | 48.5–52.2 | 0.66–0.71x |
| rowwise_int4_asym | да | 46.9–52.1 | 0.65–0.72x |
| rowwise_int4_asym | нет | 51.8 | 0.71x |
| rowwise_int4_gptq | да | 27.8–48.4 | 0.38–0.66x |

Квантованная модель **медленнее** fp16 при генерации. Triton-кернели несут оверхед на распаковку int4. На A100 такой инференс попадает в compute-bound и выигрыш от 4x меньшего объёма весов не компенсирует дополнительный compute.


### Запуск контейнера

Dockerfile совместим с драйверами cuda 12.3 или новее

В docker-compose.yaml можно пробросить желаемый порт (для triton-vis, например) или добавить команду запуска jupyter notebook

```bash
docker compose up -d
```


### Запуск тестов

```bash
pytest -sv
```
### Бенчмарки

Запуск бенчмарка для _quantize_rowwise_int4:

python triton_kernels/quantize_kernel.py
